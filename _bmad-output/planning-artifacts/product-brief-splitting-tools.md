# Product Brief：Costco 分帳小工具（splitting_tools）

## Executive Summary

一群朋友相約去 Costco，一個人刷卡付了整車兩千多塊，回到家面對一張全是商品縮寫的天書收據——沒人想算，整箱的東西誰拿幾個也吵不清，最後不是隨便均分就是不了了之。**splitting_tools 是一個給自己跟朋友用的方便小工具**：付款人拍下收據，視覺 LLM 把明細還原成看得懂的品項並自動對帳到收據總計，付款人核對無誤後產生一個免註冊連結丟進群組，朋友各自打開、認領屬於自己的品項，同一品項可多人依份額分攤（A 5 份、B 3 份），最後每個人看到自己該付多少。

這不是要打市場的產品，沒有帳本、沒有金流、沒有商業模式——就是把「Costco 完一車貨怎麼拆」這件每次都很煩的事，變成車上幾分鐘就能搞定。

**單樣本可行性（誠實標註，準確率未知）：** 已用一張真實蘇州 Costco #5564 收據驗證——解析後行加總減 3 筆 IRC 折扣＝¥2,208.50，與收據印製總計一分不差。**這是 n=1**：證明了流程可行，未證明跨收據的解析準確率。v1 須建立 3–5 張不同狀態收據的最小回歸測資。

**為什麼是現在：** 視覺 LLM 已能讀懂揉皺、有陰影的多行收據，遠勝傳統 OCR 約 85% 的天花板，且能靠「加總對帳到印製總計」自我驗證——解掉收據分帳工具最大的棄用點。但對帳只證明**總額**，不證明逐行品名與折扣歸屬正確（見下）。

## The Problem

真實場景：一群朋友去 Costco 採購，一人結帳墊全額。痛點集中在回家之後：

- **天書收據沒人想算（動機問題）**：Costco 收據是 `KS 5号电池48入`、`小金文奶酪125G*2` 這種縮寫，二三十行。墊錢的人沒道理還做苦工逐行算；其他人更不會主動。結果常是隨便均分（不公平）或拖著不了了之（傷感情）。
- **整箱誰拿幾個吵不清（時機問題）**：多件裝「你拿 10 我拿 20」，回家才算記憶已糊，沒有當下紀錄就會吵。
- **現有工具不對症**：Splitwise 逐項只能 0/50/100% 且付費牆；Tab/Splyt 要裝 App、要先邀朋友；Tricount/Splid 零註冊但不解析收據。沒有一個對準「量販一車混合貨、免註冊、整箱按份額拆」。

## The Solution

一條從拍照到分完的低摩擦流程，發起人一個人就能在車上完成：

1. **拍照解析**：付款人拍收據（上傳前自動降尺寸壓縮，省流量與延遲）→ 單次視覺 LLM 呼叫還原品名、IRC 即時折扣配對到母品項。
2. **發起人核對關卡**：不只比對總額。系統呈現 (a) 解析加總 vs 印製總計差額、(b) **被標記需肉眼確認的可疑行**（單行佔比過高、品名互換疑慮）、(c) **每筆 IRC 折扣歸到哪個母品項，一鍵改綁**。可編輯金額/品名/折扣歸屬/增刪行。
   - **閘門有底、有逃生口**：印製總計糊掉/被遮時，付款人可手動輸入收據上的總計；連續 N 次仍兜不攏可「未驗證強制放行」，並對所有認領者顯示「未驗證」橫幅。閘門永不把人卡死在停車場。
3. **產生免註冊連結**：核對後產生不可猜連結；輸出一段**可貼的訊息卡**（含日期/總額/品項數/付款人，非裸網址）丟進既有群組，避免被當成可疑連結。
4. **朋友自助認領**：點開連結，**身份綁定到本機 token**（同一裝置再開仍是同一人，避免撞名/錯字/重複分裂；進入時可從既有名單挑「是不是你」）。勾選或剔除自己的品項；**每人只能改自己這台裝置的認領**，付款人可合併/改名/仲裁。每人可按「我認領完了」。
5. **彈性分攤**：同一品項可多人共分，預設均分，可改份額（A:5／B:3，沿用 Splitwise「份數」心智；10:20 拿量比即 shares 1:2，v1 加權份額已可表達整箱比例）。
6. **未認領＝待處理（非靜默歸付款人）**：未被認領的品項標為 PENDING，付款人看到「N 項／¥Z 尚未有人認領，目前算你的」即時橫幅；**必須由付款人主動點「結束分帳並吸收剩餘 ¥Z」才定案**——不靠超時或靜默 default，避免「沒人來認領 → 系統顯示已結清 → 付款人默默吃下全車」這個把公平性反轉的最常見路徑。
7. **結算頁**：每人付自己認領各行金額之和（VAT 已內含每行，無需獨立稅金分攤——**僅 #5564 驗證的假設**，見 Open Questions）。頁面：極簡「誰欠誰多少」、一鍵**複製可貼回群組的純文字結算**、顯示「✓ 已對得上收據總計 ¥X」信任標記給認領者看、原收據縮圖可點開查「這啥」、付款人剩餘額明示為「其餘 N 項預設為付款人的」（非故障感）。

## 設計取捨（誠實版，非護城河）

個人工具，不主張技術護城河。刻意取捨：

- **OCR 不是難點，「可信」才是**：對帳閘門幾乎免費，是「能斷定解析這次對了」的尺；但它只證明總額，**逐行品名/折扣歸屬錯誤它擋不住**——所以閘門＝總額對帳＋付款人對可疑行與 IRC 歸屬的肉眼抽檢，缺一不可。
- **減法優先**：明確不做護城河、長期帳本、金流收款、地理圍欄、語音、遊戲化、跨幣別報帳。
- **借成熟模式不重造輪子**：免註冊連結（Tricount）、份數分攤（Splitwise）、認領看板（WeChat 群收款）皆已驗證。

## Who This Serves

**主要使用者——發起人（付款人）**：揪團、刷卡墊全額的人。要的是「不用做苦工、金額一定對、朋友會配合、剩餘額被當成我的時我心裡有數」。Aha：拍完照看到天書變看得懂的明細且兜對總計。

**次要使用者——認領的朋友**：被揪去一起買的人。要的是「不裝 App、不註冊、點開 30 秒勾完」。Aha：打開只要勾自己那幾樣。

兩者都在「結帳完、人還在賣場/車上」這個時機使用——趁人都在、記憶最新。

## Success Criteria（可觀察的單次 dogfood 通過清單，非商業指標）

一次真實揪團 Costco，當場跑這套，逐項判定通過/失敗：

- 全程在賣場/車上完成，**零事後群組補算**。
- 付款人**無需手動逐行修正超過 ~3 項**即可通過核對關卡。
- 結束時**沒有「未認領被靜默吃掉」**：付款人是有意識地按下吸收剩餘，或所有他人項目都被認領。
- 朋友端從點開連結到「我認領完了」**無需任何口頭解釋**。
- 解析總額對帳通過（這是設計保證、非成果指標，僅作前置條件）。

> 註：拍照→解析含一次外部視覺 LLM 呼叫，弱網下常 8–25s。依 side-project 標準（>1s 須 async），v1 採「立即回 job_id + 輪詢進度」而非阻塞等待；不設不切實際的「<1 分鐘」硬指標。

## Scope

**v1 In（POC，限 Costco，#5564 為固定回歸測資並寫成可執行斷言 `parsed_sum == 2208.50`）：**
1. 拍照（前端壓縮）→ 視覺 LLM 解析（job_id + 輪詢）→ IRC 配對
2. 發起人核對關卡（總額＋可疑行＋IRC 歸屬；有底、有逃生口）
3. 核對後產生不可猜連結 + 可貼訊息卡
4. 免註冊、本機 token 綁身份；每人只能改自己的；付款人仲裁/合併/改名
5. 單品項多人共分，預設均分、可改份額（A:5／B:3）
6. 未認領＝PENDING；付款人顯式定案才結清
7. 結算頁：誰欠誰 + 一鍵複製文字 + 對帳信任標記 + 收據縮圖 + 付款人剩餘明示
8. 同團名字本機記憶（localStorage，下次免再打）

**v1 Out：** 護城河/差異化、長期帳本、金流/收款、地理圍欄、語音、遊戲化、跨幣別報帳、非 Costco、websocket 真同步（用輪詢）、按數量拆專屬 UI（#13，份額已可替代表達，v2 再做專屬）。

## Resolved Decisions（已拍板 2026-05-18）

1. **稅金模型**：v1 採「無獨立稅金分攤（VAT 內含每行）」。**硬鎖只接受 #5564 同結構收據；偵測到不同結構（獨立稅行/不同稅碼佈局/幣別）即明確拒絕**，不靜默誤算。Fallback 規則（行加總≠含稅總計時按認領小計比例攤稅）記錄於 v2 backlog，v1 不實作。
2. **未認領＝PENDING（修正原決策1）**：未認領品項標 PENDING，**付款人必須主動點「結束分帳並吸收剩餘 ¥Z」才定案**；不靠超時/靜默 final。此正式取代原「未認領自動歸付款人」之靜默行為。
3. **身份＝本機 token**（取代名字字串，為原 #21 認領完整性之定案解）：同裝置再開仍同一人；每人只能改自己裝置的認領；付款人可仲裁/合併/改名。
4. **資料處理**：上傳前**遮蔽會員卡號**；結清後 session **凍結**（唯讀）；分享連結 **30 天到期**。收據影像/claims 僅保留至到期即銷毀。

## Roadmap Thinking（grounded）

v1 自己跟朋友用順手後，v2 候選（依實際 dogfood 痛點再決定）：按數量拆專屬 UI（#13）、即時看板真同步、結算頁自動最少轉帳次數、非 Costco 收據通用化。金流/帳本仍刻意排除。

## 開發前提醒（給後續 PRD / 架構）

- OCR＝視覺 LLM 外部 API 邊界，須套 side-project 7 項非協商標準；v1 最小集：指數退避重試、結構化 log（model/tokens/latency/cost/session_id）、成本追蹤須**持久化**（Postgres，非記憶體）、**parse 端點 per-session/per-IP 預算上限**（免註冊開放連結防濫用）、>1s 走 job_id+輪詢。
- 連結＋多人認領狀態需輕量後端（單一全端服務 + Postgres 存 session/claims）；不可猜連結 ID + 結清後凍結 + session 到期。
- 身份＝本機 token（非名字字串）；認領變更需有 undo 與付款人可見變更紀錄。
