## Phase 1: 擴張探索 — What If Scenarios

### ⚓ 核心情境錨點（使用者親述的真實痛點）

> 「跟朋友一起去 Costco 買東西，拆帳起來很麻煩。」

**重大重構：** 主場景不是餐廳分帳，而是 **Costco / 量販店「一車混合貨、一人結帳、事後多人拆」**。與餐廳分帳的關鍵差異：
- 品項是實體商品，常為「整箱/多件裝」需按數量比例拆（30 條燕麥棒你拿 10 我拿 20）
- 一人刷卡付整車 → 天生的「債主 vs 欠款人」結構（呼應 WeChat 群收款）
- Costco 收據是縮寫天書（如 `KS ORG EGGS 24CT`），OCR + LLM 還原語意 = 真正護城河
- 事後對著天書收據回想「這啥、誰買的」是最大摩擦點

### 💡 Phase 1 點子捕捉

**[功能 #1]**：逛場邊買邊分（Buy-and-Tag）
_Concept_：放棄事後認領，推車時每拿一件就快速標記「我的 / 50-50 / 這箱分 4 份」，結帳前就分完。
_Novelty_：把分帳從「事後考古」變成「現場順手」，繞開 Costco 天書收據問題。

**[功能 #2]**：Costco 天書翻譯機（OCR + LLM 語意還原）
_Concept_：把 `KS ORG EGGS 24CT` 還原成「科克蘭有機蛋 24入」並自動猜分類。
_Novelty_：一般分帳 App（Splitwise）完全做不到量販縮寫還原，這是差異化護城河。

**[功能 #3]**：整箱按數量拆（Quantity-aware Split）
_Concept_：對多件裝商品支援「件數比例」分攤，而非只有均分。
_Novelty_：分帳 App 普遍只做「均分/百分比」，缺「同一品項按拿取數量拆」。

**[商業/結算 #4]**：債主自動收款（WeChat 群收款模式）
_Concept_：一人付整車後，App 自動生成「你欠長安 NT$430」並串一鍵轉帳。
_Novelty_：把分帳結果直接接到收款閉環，而非算完丟著沒人付。

### 🌟 BREAKTHROUGH #1：停車場分帳模式（Parking Lot Mode）

> 使用者親口認證「真的是太神來一筆」。

**核心洞察：** 拆帳爭議（痛點 B+D）的根本不是算法不公或 OCR 不準，而是 **時機錯位**——回家才算，記憶已糊。解法是把分帳塞進「結帳完、後車廂分貨、還沒鳥獸散」那關鍵 3 分鐘。

**[體驗 #5]**：停車場 3 分鐘閃分
_Concept_：結帳後在後車廂分貨當下完成歸屬，趁記憶最新、人都還在。
_Novelty_：競品都假設「事後對帳」，這是唯一抓住「分貨當下」的分帳產品。

**[行為 #6]**：地理圍欄時機推播
_Concept_：偵測到你剛離開 Costco，自動跳「剛結帳了嗎？要開分嗎？」的 nudge。
_Novelty_：把「沒人想發起」的動機問題用時機推播解決，不靠人記得。

**[痛點 B #7]**：付款人零勞動 + 他人 1-tap 確認
_Concept_：AI 翻譯天書收據+預設分法，付款人只按送出；其他人滑確認自己的，套 WeChat「已確認/未確認」社交壓力。
_Novelty_：勞動從債主轉嫁到「想省錢的人」，靠社交壓力收尾。

**[痛點 D #8]**：整箱即時共享計數器 + 拆貨存證照
_Concept_：多件裝顯示即時計數，分貨時各自 +1；分完拍「N 堆」照存證。
_Novelty_：在分貨物理動作當下捕捉歸屬，杜絕事後回憶爭議。

**[痛點 D #9]**：「我沒拿」反向否決制
_Concept_：預設整箱均分，只有主動喊「這箱我沒碰」才扣除，舉證責任反轉。
_Novelty_：用預設值消滅 99% 的逐件確認摩擦。

### 🌟 BREAKTHROUGH #2：自我對帳驗證（Self-Reconciling Parse）— 已用真實收據驗證

> 使用者丟真實蘇州 Costco #5564 收據（總計 ¥2,208.50）實測。OCR 還原 28 行（含 3 筆 IRC 折扣），加總正項減折扣 = ¥2,208.50 與印製總計一分不差。可行性已驗證。

**[信任 #10]**：對帳閘門（Reconciliation Gate）
_Concept_：解析後自動 sum 比對收據印的「總計」，不符就標紅請使用者補拍/修正，相符才放行分帳。
_Rationale（誠實修正）_：定位**不是**「因為視覺 LLM 不可靠」——實測這張含陰影/遮擋的真實收據，LLM 解析完全正確。對帳閘門的真正價值是：(1) 它正是「能斷定 LLM 這次對了」的那把尺（使用者對解析的信心來自加總=¥2,208.50，而非盲信模型）；(2) n=1 不代表可靠率，未測情境含熱感紙褪色/過曝反光/超長收據中段模糊；(3) 實作近乎免費（幾行 sum 比對），平時不擋路、僅兜不攏時提示，而錯帳代價是朋友間社交尷尬。→ POC 必做，但理由是「免費的正確性尺」非「補 LLM 弱點」。

**[痛點 B #11]**：IRC 折扣自動配對母品項
_Concept_：辨識 `IRC #商品碼` / `9029xxx` 折扣行，自動歸到對應母商品淨價（如威士忌類折扣只折給拿酒的人）。
_Novelty_：Costco 即時折扣（IRC）綁特定商品，亂攤就不公平；自動配對是量販分帳專屬難題。

**[資料 #12]**：品名正規化（縮寫/在地化還原）
_Concept_：LLM 把 `KS 5号电池48入`、`小金文奶酪125G*2`、`滑芙蜂蜜豆浆` 還原成可讀品名 + 分類 + 內含數量。
_Novelty_：解析「名稱內嵌數量（×2 / 48入）」與品牌縮寫（KS=Kirkland），一般 OCR 不懂。

**[痛點 D #13]**：名稱內嵌數量 → 自動觸發按量拆
_Concept_：偵測到 `125G*2`、`48入` 這種多件裝，自動跳「這箱要按人頭分數量嗎？」
_Novelty_：把痛點 D（整箱誰拿幾個）從使用者要記得設定，變成系統主動偵測提示。

**[資料 #14]**：稅碼/區段結構保留（A/B/Z、底層商品）
_Concept_：保留每行稅碼與「底層商品」（推車底層大件掃描區）等結構，供日後跨幣別/發票需求。
_Novelty_：多數分帳 App 丟掉稅務結構；保留可支援報帳、跨境（蘇州店為人民幣含 VAT）情境。

### 🧭 SCOPE PIVOT（使用者明確指示 — 最高優先）

> 「妳不需要想護城河，我只想要一個方便的小工具。」

**範圍重新定義：** 這**不是**要打市場的產品，是**個人/朋友間用的方便小工具**。下列方向標為 **OUT OF SCOPE**（過度工程，不做）：
- ❌ 護城河 / 差異化 / 對標 Splitwise（#2 護城河論述、Breakthrough 的「競品輸在…」）
- ❌ 長期滾動帳本 / 關係黏著（#7、原 #4 商業閉環的延伸）
- ❌ 地理圍欄推播（#6）、語音 AI 分帳、遊戲化搶貨、商業模式
- ❌ 跨幣別報帳 / 發票稅務延伸（#14 降級為「保留欄位即可，不做功能」）

**仍 IN SCOPE（小工具核心）：**
- ✅ 拍 Costco 收據 → OCR 還原品名 + 自我對帳到總計（Breakthrough #2，已驗證可行）
- ✅ 共享品項支援「分 N 人」或「按數量拆」（#3、#13）
- ✅ IRC 折扣自動歸對母品項（#11 — 真單裡 3 筆，不處理會算錯）
- ✅ 算出每人該付多少，產出一份好分享的結果
- ✅ 結帳後當下就能用、免建群組/帳號的低摩擦流程（停車場模式的「快」精神保留，但不做花俏版）

### 🔧 REFINEMENT（使用者細化 — 修正先前過度簡化）

> 「妳可以參考其他競品。我的想法是丟一個連結大家自己勾或剔除，然後能看到進來的人點了哪些？然後一個品項可以不只一個人分，同一品項還可以 A 分 5、B 分 3 這樣。」

**修正：** 先前我把方案簡化成「單人操作、無後端、截圖貼群」——**使用者否決**，正式採用 **連結協作模型**。需輕量後端保存共享狀態（非過度工程，是真實需求）。

**[核心模型 #15]**：連結共享 + 自助勾選/剔除
_Concept_：付款人建立後丟一個連結，其他人點開自行勾選（認領）或剔除品項，免註冊免裝 App。
_Novelty_：借 Tricount/Splid 的免帳號連結模式；認領由本人做，非付款人代勞。

**[核心模型 #16]**：即時認領看板（誰進來了、點了什麼）
_Concept_：所有人即時看到誰已加入、各自已勾哪些品項，未認領的品項一目了然。
_Novelty_：借 WeChat 群收款/Venmo 的「狀態可見」社交壓力，收尾不靠人催。

**[核心模型 #17]**：單品項多人 + 加權份額（A:5 / B:3）
_Concept_：同一品項可多人共分，且支援自訂份額比例（非僅均分），按權重攤金額。
_Novelty_：借 Splitwise「by shares」成熟模式；解決真實情境（同箱拿不一樣多）。

**[核心模型 #18]**：發起人確認關卡（Confirm-before-Share）
_Concept_：解析後先進發起人確認流程——系統自動對帳結果輔助，發起人核對/修正品項與總額，**確認無誤才產生並分享連結**。Session 生命週期：草稿 → 已確認 → 分享 → 認領中 → 結清。
_Novelty_：擋住「解析錯卻已分享 → 全員對錯資料認領」的大坑；同時把對帳閘門（#10）定位收乾淨為「輔助發起人確認」而非自動擋人。

**競品參考（借成熟做法，非對標）：** Splitwise by-shares 分攤 · Tricount/Splid 免帳號連結 · WeChat 群收款/Venmo 即時狀態看板。

**架構含意（給下一步 PRD/架構）：** 連結 + 即時多人狀態 → 需輕量後端（建議 Postgres 存 session/claims，單一全端服務即可，DAU 未達 10k 不引入 microservices）。OCR 步驟為 LLM/視覺 API 呼叫 → 屆時須套用 side-project 7 項非協商標準（重試退避、成本追蹤、>1s 走 async、降級鏈、結構化 log 等）。

**Theme 3 — 結算輸出**

**Breakthrough（保留精神）**

- BREAKTHROUGH #2 自我對帳 → 已用真實收據驗證（¥2,208.50 一分不差），列 POC 核心

**Out of Scope（記錄但明確不做）**
- 護城河/差異化論述 · 長期滾動帳本 · 地理圍欄推播 · 語音分帳 · 遊戲化搶貨 · 商業模式 · 跨幣別報帳/發票

### 優先排序（POC 範圍）

**Must-have（POC，限 Costco）：**
1. 拍照 → 視覺 LLM 解析明細
2. **發起人確認關卡**：系統自動對帳（加總 vs 收據總計）輔助 → 發起人核對/修正品項與總額 → 確認無誤才放行
3. 確認後產生分享連結
4. 連結打開輸入名字（免註冊）
5. 未認領自動歸付款人；他人勾選認領/剔除
6. 單品項多人共分，預設均分、可改加權份額（A:5/B:3）
7. 算每人應付（IRC 折扣歸位）+ 可分享結果頁

**Defer（POC 後 v2）：** 即時同步看板（POC 先用重整/輪詢，不做 websocket）· 按數量拆（#13）· 稅碼結構僅存不顯示

### 行動計畫（Next Steps）

1. **技術骨架**：單一全端服務（建議 Next.js）+ 輕量 DB（POC 可 SQLite/Postgres）存 session 與 claims；視覺解析用 Claude vision 單次呼叫；POC 以 Costco #5564 收據為固定測資。
2. **LLM 邊界最小防護**（POC 階段最小集）：指數退避重試 + 結構化 log（model/tokens/latency/cost）+ 每次呼叫記一筆成本；async/queue 與降級鏈列 v2。
3. **進 BMAD 下一關**：把這份定義轉成正式 spec —— 建議 `[CB] bmad-product-brief` 收斂產品輪廓，或想直接動工則 `[CP] bmad-create-prd`。建議在全新 context window 執行。
